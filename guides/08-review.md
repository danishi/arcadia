# Phase 7: Quality Review Guide

RFP適合性の最終検証、数値整合性チェック、クロスリファレンス確認を行うフェーズ。

---

## Overview

| Attribute | Value |
|-----------|-------|
| Phase | 7. Review |
| Duration | 継続的に実施; 提出前1-2日で集中実施 |
| Human/AI Split | 10% / 90% |
| Primary Actor | AI (Claude Code) |
| Parallelizable | Yes -- 全フェーズと並行して継続的に実施 |

提案成果物全体がRFP要件を漏れなくカバーしているか、数値に矛盾がないか、ボリューム間の整合性が取れているかを機械的に検証する。AIが体系的なスキャンと修正提案を行い、人間が最終的なGo/No-Go判断を下す。

## Human Tasks

人間が判断・実行すべき作業:

- [ ] 監査レポートを確認し、MISMATCH/MISSING項目の対応方針を判断する
- [ ] 対応不要と判断した項目に理由を付記する
- [ ] 最終版の提案書・回答シートを通読レビューする
- [ ] プレゼンリハーサルを実施する
- [ ] 提出前の最終Go/No-Go判断を行う
- [ ] 提出物の一覧・ファイル名・フォーマットがRFP指定に合致しているか確認する

## Claude Code Instructions

以下の指示文をClaude Codeにコピペして使用する。

### 1. RFP適合性の全体チェック

```
/rfp-auditor で提案書全体のRFP適合性をチェックして。
rfp-requirements-checklist.md の全要件について:
- MATCH: 提案書で適切に対応されている
- MISMATCH: 提案書の記載がRFP要件と矛盾している
- MISSING: 提案書で対応されていない
判定結果とカバレッジ率を報告して。
```

### 2. 数値整合性チェック

```
/rfp-auditor で見積の数値を原典と照合して。
以下を確認:
- estimation-policy.md 内の数値が自己矛盾していないか
- 提案書内の金額が estimation-policy.md と一致しているか
- 回答シート（XLSX）の数値が提案書と一致しているか
- Ph1/Ph2の合計が総額と一致しているか
```

### 3. クロスリファレンス整合性チェック

```
提案書Vol1-Vol6のスライド設計書を読んで、以下をチェックして:
- ボリューム間で矛盾する記述がないか
- 同じ事項を異なる数値・表現で記載していないか
- 参照先（「Vol3参照」等）が正しいか
- スコープ（Ph1/Ph2）の記載が全ボリュームで一致しているか
```

### 4. 戦略整合性チェック

```
proposal-strategy.md のWinテーマ・差別化ポイントが、提案書全体を通じて一貫して訴求されているか確認して。
Winテーマごとに、どのボリュームのどのスライドで言及されているかマッピングして。
```

### 5. デモと提案書の整合性チェック

```
提案書Vol6（デモ編）の記載内容と、実際のデモアプリ（src/）の画面構成・機能が一致しているか確認して。
差異があれば指摘して。
```

### 6. 提出物一覧の最終確認

```
RFP本紙の提出物要件を読み取り、RFP_answer/ ディレクトリの内容と照合して。
不足しているファイル・フォーマット違反があれば指摘して。
```

### 7. 指摘事項の一括修正

```
監査レポートのMISSING/MISMATCH項目のうち、AI側で修正可能なものを修正して。
修正内容を報告して。
```

## Checklist

Phase完了の判定基準:

- [ ] `rfp-auditor` による全体スキャンが完了し、監査レポートが生成されている
- [ ] RFP要件カバレッジ率が目標値（95%以上推奨）に達している
- [ ] MISMATCH項目がゼロ、またはすべて是正済みである
- [ ] MISSING項目について、対応済みまたは「対応不要」の判断が記録されている
- [ ] 見積数値が全成果物間で一致している
- [ ] ボリューム間のクロスリファレンスに矛盾がない
- [ ] Winテーマが提案書全体を通じて一貫して訴求されている
- [ ] デモアプリと提案書の記載が整合している
- [ ] 提出物一覧がRFP指定に合致している
- [ ] 人間による最終レビュー・Go/No-Go判断が完了している
- [ ] プレゼンリハーサルが実施されている（プレゼンがある場合）

## Review Timing

品質チェックは以下の3段階で実施する:

### Stage A: 中間チェック（Phase 3-6 並行期間中）

各フェーズの成果物が完成するたびに、個別の整合性チェックを実施する。

```
architecture-policy.md がRFP技術要件をカバーしているか簡易チェックして
```

### Stage B: 統合チェック（全成果物揃い次第）

提案書・見積・設計書・デモが揃った段階で、クロスリファレンスを含む統合チェックを実施する。

```
提案書全体の統合チェックを実施して（適合性 + 数値整合性 + クロスリファレンス）
```

### Stage C: 最終チェック（提出前日）

提出物一覧の確認、ファイル名・フォーマットの最終確認を行う。

```
RFP_answer/ の提出物一覧をRFP要件と照合して、最終チェックリストを生成して
```

## Tips

- Phase 7は「最後にまとめて実行」ではなく、Phase 3以降で継続的に実施する
- 最も見落としやすいのは「ボリューム間の数値不整合」。見積額の変更時は全ボリュームへの波及確認が必須
- MISSINGの判定は保守的（見落としより誤検知が多い）。人間が「対応不要」と判断したものには理由を記録する
- 提出直前のフォーマットチェック（ファイル名、拡張子、ページ数制限等）は意外と重要。RFPの提出要件を原文で確認する
- プレゼンリハーサルでは、デモの動作確認と提案書のストーリーラインの通し確認を同時に行う
