# 提案書スライド設計書ジェネレーター

## 概要

本スキルは、提案書作成の**3層構造**に基づき、スライド設計書（Markdown形式）を生成・更新する。
提案戦略からスライド1枚1枚のメッセージまでを一貫した論理で繋ぎ、抜け漏れのない提案書構成を実現する。

### 3層構造

```
提案戦略書（Why: なぜ勝てるか）
  ↓ Win Strategy / 差別化の柱を反映
チェックリスト（What: 何を書くか）
  ↓ Vol別の章立て・必須項目を網羅
スライド設計書（How: どう伝えるか）
  → 各スライドのメッセージ・内容・スピーカーノート
```

---

## スライド設計書生成手順（5ステップ）

### Step 0: 組織固有データの読込

`arcadia/org-data/` 配下の自社固有データを読み込み、提案書に活用可能な自社情報を把握する:

- **`company-profile.md`**: 会社概要、認証・資格、パートナーシップ、導入実績（実績IDで引用）
- **`service-catalog.md`**: 自社サービスの仕様・差別化ポイント（サービスIDで引用）
- **`whitepapers/index.md`**: 引用可能なホワイトペーパー（タグ別・章別の引用推奨マップ）

> org-data が未整備の場合でもスライド設計書の生成は可能だが、
> 会社紹介・実績・差別化セクションの内容が汎用的になるため、整備を推奨する。

### Step 1: 提案戦略書読込

提案戦略書（`output/plan/proposal-strategy.md` 等）を読み込み、以下を把握する:

- **Win Strategy**: 提案全体の勝ちストーリー
- **差別化の柱**: 競合との差別化ポイント（3-5本程度）。org-data/service-catalog.md の差別化ポイントと整合させる
- **キーメッセージ**: 顧客に最も伝えたい価値提案
- **リスクと対策**: 提案上の弱みとそのカバー方針
- **共同提案体制**: パートナー各社の担当範囲。org-data/company-profile.md のパートナーシップ情報と整合させる

### Step 2: チェックリスト読込

提案書作成チェックリスト（`output/plan/proposal-items-checklist.md` 等）を読み込み、以下を把握する:

- **Vol別の章立て**: 各分冊の構成と必須セクション
- **必須記載項目**: RFPが指定する記載必須事項
- **想定スライド数**: 各Volの目安ページ数
- **重点項目**: 配点が高い/審査で重視される項目

### Step 3: スライドメッセージ設計

**1スライド1メッセージ原則**に基づき、各スライドを設計する:

- 各スライドに1つの明確なメッセージ（「このスライドで何を伝えるか」）を設定
- メッセージは提案戦略書のWin Strategyと紐づける
- 情報過多のスライドは分割を検討
- ストーリーの流れ（起承転結）を意識した並び順にする

### Step 4: Markdown形式でスライド設計書を出力

以下のフォーマットで出力する:

```markdown
# Vol.X: [分冊タイトル]

## メタ情報
- **想定スライド数**: XX枚
- **対象チェックリスト項目**: [項目ID一覧]
- **キーメッセージ**: [このVolで最も伝えたいこと]

---

## スライド一覧

### Slide 1: [スライドタイトル]
- **メッセージ**: [このスライドで伝えたい1つのこと]
- **内容**:
  - [箇条書きで記載内容を列挙]
  - [図表がある場合はその概要も記載]
- **スピーカーノート**: [プレゼン時の補足説明・強調ポイント]
- **対応チェックリスト**: [項目ID]

### Slide 2: [スライドタイトル]
...
```

### Step 5: チェックリストとの突合

生成したスライド設計書とチェックリストを突合し、以下を確認する:

- **漏れ確認**: チェックリストの全必須項目がいずれかのスライドでカバーされているか
- **重複確認**: 同じ内容が複数のVolで不必要に重複していないか
- **整合性確認**: Vol間でメッセージが矛盾していないか

突合結果をスライド設計書の末尾に付記する:

```markdown
## チェックリスト突合結果
| チェック項目ID | 対応スライド | 状態 |
|--------------|------------|------|
| E-1-1 | Vol.1 Slide 3 | カバー済 |
| E-1-2 | - | 未対応（要追加） |
```

## Vol分冊構成の決定プロセス

分冊構成は固定ではなく、**RFP要件・コンテンツ量・可読性**を考慮してプロジェクトごとに設計する。
以下の判断基準に基づき、提案戦略書（`proposal-strategy.md` セクション3）で分冊構成を定義する。

### 判断基準

#### 1. RFPの指定（最優先）

RFPが提案書の構成・分冊方法を明示している場合、その指定に従う:

- 「技術提案と費用提案を分冊すること」等の明示的指定
- 「○○について別紙で回答すること」等の別紙要求
- 提出物一覧で分冊が暗示されている場合（例: 技術評価書と価格評価書が別テーブル）
- ページ数上限の指定がある場合はその制約内で構成

#### 2. LLMコンテキストサイズの考慮

AIがスライド設計書を生成・レビューする際の品質を維持するため:

- **1 Vol あたりの目安**: 20-40スライド（Markdown設計書で約3,000-6,000行相当）
- 40スライドを超える場合は分割を検討する（コンテキスト内での一貫性維持が困難になるため）
- 逆に、10スライド未満の分冊は他Volとの統合を検討する
- クロスリファレンスチェック時にVol全体を一度に読み込めるサイズが望ましい

#### 3. 可読性・ストーリー性

評価者（人間）が読みやすい構成にする:

- **関連する内容はまとめる**: 技術構成と移行計画は密接に関連するため、同一Volまたは連続Volに配置
- **評価者の関心で分ける**: 技術評価者が読むVolと、経営層が読むVolを意識する
- **自己完結性**: 各Volが単体でも主要メッセージを理解できる構成にする（ただし冗長な重複は避ける）
- **ストーリーの流れ**: 課題→解決策→実現計画→投資 の論理的順序を保つ

### 分冊数の目安

| 提案書総量 | 推奨分冊数 | 備考 |
|-----------|-----------|------|
| 30枚以下 | 1冊（分冊不要） | 小規模RFP、RFI回答等 |
| 30-80枚 | 2-3冊 | 技術/管理/費用 程度の粗い分割 |
| 80-150枚 | 3-5冊 | 標準的なRFP対応 |
| 150枚超 | 5-8冊 | 大規模案件、複数フェーズ提案 |

> 上記はあくまで目安。RFP指定がある場合はそちらに従い、ない場合もプロジェクト特性に応じて柔軟に決定する。

### 典型的な分冊パターン（参考）

RFPの指定がない場合に参考とする典型パターン。プロジェクトに応じて取捨選択・統合・分割する:

| パターン | 想定内容 |
|---------|---------|
| 概要・会社紹介 | エグゼクティブサマリー、会社概要、実績、提案コンセプト |
| 技術提案 | システム構成、アーキテクチャ、機能要件対応、データ連携・移行 |
| プロジェクト計画 | スケジュール、体制、開発方法論、品質管理、教育 |
| 費用 | イニシャル/ランニング費用、TCO、費用内訳 |
| 運用・保守 | SLA/SLO、監視、障害対応、運用体制 |
| 付録 | 制約事項、法務、デモ説明、補足資料 |

> これは「6分冊にせよ」という意味ではない。例えば30枚の提案書であれば「概要+技術」「計画+費用」の2冊で十分な場合もある。

## 出力先

| 成果物 | パス | 説明 |
|--------|------|------|
| スライド骨子 MD | `output/slides/vol{N}-{topic}/slides.md` | 本スキルの主出力 |
| デザイン指示 MD | `output/slides/vol{N}-{topic}/design.md` | NanoBanana用のビジュアルデザイン指示 |
| スライド画像 | `output/slides/vol{N}-{topic}/slide-{NN}.png` | NanoBananaスキルが生成 |

> Volume別サブフォルダに骨子・デザイン・画像を集約する。`{topic}` は kebab-case（例: `vol1-overview`, `vol2-technical`）。

---

## 出力形式

### md（デフォルト）

上記Step 4のフルフォーマットで出力する。スライド制作担当者がそのまま作業に使える詳細度。

### outline

簡易アウトライン形式で出力する。構成レビュー用。

```markdown
# Vol.X: [分冊タイトル]（XX枚）

1. [スライドタイトル] - [メッセージ]
2. [スライドタイトル] - [メッセージ]
3. ...
```

---

## 注意事項

- スライドタイトルは顧客目線で価値が伝わる表現にする（技術用語の羅列を避ける）
- スピーカーノートには、スライドに書かない「裏の意図」や「想定質問への回答」も含める
- 提案戦略書が存在しない場合は、まず提案戦略書の作成を推奨する
- チェックリストが存在しない場合は、RFP本紙から必須記載項目を抽出して仮チェックリストを作成する
- org-data の実績（REF-XXX）やサービス（SVC-XXX）を引用する場合は、IDを明記して追跡可能にする
- ホワイトペーパーを引用する場合は whitepapers/index.md の引用ガイドラインに従う
