# references/ ディレクトリ

本ディレクトリには、rfp-auditorスキルが参照するプロジェクト固有の参照ファイルを格納する。
これらのファイルは**案件開始時にClaude Codeが対話的に生成**する。

---

## 生成すべきファイル

### 1. rfp-requirements-checklist.md

RFP本紙から抽出した要件項目を、チェック可能な粒度に分解したチェックリスト。

#### 構造

```
# RFP要件チェックリスト

## A. 機能要件（必須提案範囲）
### A-1. [機能カテゴリ名]
[エビデンス: `docs/配下の参照先パス`]
- [ ] A-1-1: [要件概要]
  → [確認すべき原典ファイルパス]
- [ ] A-1-2: ...

## B. 機能要件（オプション/参考価格範囲）
...

## C. 非機能要件
...

## D. インターフェース要件
...

## E. 提案書構成要件
...

## F. デモ・PoC要件
...

## G. 重要ビジネス数値
...
```

#### 生成手順

1. RFP本紙（`docs/` 直下のRFPファイル）を全文読み込む
2. 要件記述を抽出し、カテゴリA-Gに分類する
3. 各要件を検証可能な粒度に分解する（1項目=1検証ポイント）
4. 各要件に `[エビデンス]` タグを付与し、参照すべき原典資料のパスを記載する
5. チェックボックス形式（`- [ ]`）で出力する

#### 生成コマンド例

```
Claude Codeへの指示:
「docs/配下のRFP本紙を読み込み、rfp-auditorスキルのreferences/rfp-requirements-checklist.md を生成してください。
カテゴリA-G構造で要件を分類し、各要件には検証に必要な原典資料のパスを[エビデンス]タグで付与してください。」
```

### 2. docs-catalog.md

`docs/` 配下の全資料を分類し、要件チェック時にどのファイルを参照すべきかのマッピングを提供するカタログ。

#### 構造

```
# docs/ ドキュメントカタログ

本ファイルは docs/ 配下の全資料を分類し、RFP要件チェック時にどのファイルを参照すべきかのマッピングを提供する。
総ファイル数: 約XXX件

## 1. RFP本紙・提案戦略
| ファイル | 内容 | 関連要件 |
|---------|------|---------|
| ... | ... | ... |

## 2. [カテゴリ名]
| ファイル/ディレクトリ | 内容 | 関連要件 |
|---------|------|---------|
| ... | ... | ... |

...

## 逆引きマップ: 要件ID → 参照ファイル
| 要件カテゴリ | 主要参照先 |
|------------|-----------|
| A（機能要件） | ... |
| B（オプション機能） | ... |
| C（非機能要件） | ... |
| D（IF要件） | ... |
| E（提案構成） | ... |
| F（デモ要件） | ... |
| G（ビジネス数値） | ... |
```

#### 生成手順

1. `docs/` 配下を再帰的に走査し、全ファイル・ディレクトリを列挙する
2. ファイルをカテゴリ別に分類する（設計書種別、資料種別等）
3. 各ファイルの内容をサマリ化する（先頭数行の読み込みやファイル名から推定）
4. rfp-requirements-checklist.md の要件IDとの対応関係（逆引きマップ）を作成する
5. 総ファイル数を記載する

#### 生成コマンド例

```
Claude Codeへの指示:
「docs/配下を走査し、rfp-auditorスキルのreferences/docs-catalog.md を生成してください。
全ファイルをカテゴリ別に分類し、要件ID（A-G）との逆引きマップも作成してください。」
```

---

## 注意事項

- これらのファイルはプロジェクト固有の内容を含むため、テンプレートとしてはこのREADMEのみを提供する
- 生成後は内容をレビューし、分類の正確性やエビデンスパスの妥当性を確認すること
- RFPの追加資料を受領した場合は、docs-catalog.md を更新すること
- `.gitkeep` ファイルは生成ファイルが存在しない状態でもディレクトリをGit管理下に置くために配置している
